**Researched:** 2026-03-18
**Domain:** Claude Code Skill modification -- LaTeX body text generation with citation integration and bilingual output
**Confidence:** HIGH

## Summary

Phase 16 adds Step 5 (Body Generation) to the existing `repo-to-paper-skill`, which currently has Steps 1-4 plus Step 2.5 (381 lines). Step 5 generates per-H1-section LaTeX `.tex` files with `[SOURCE: file:line]` repo data annotations, `\cite{key}` inline citations sourced strictly from `.paper-refs/` files, anti-hallucination placeholders, and paragraph-by-paragraph bilingual output in LaTeX comment format. The output directory is `{repo_path}/.paper-output/`. After all selected sections are confirmed, a `references.bib` file is auto-generated by extracting and deduplicating BibTeX blocks from `.paper-refs/*.md` files.

The primary technical challenge is the SKILL.md line budget. At 381 lines (already exceeding the 300-line convention), adding Step 5 with its generation rules, citation integration, anti-hallucination logic, bilingual formatting, and `references.bib` generation will push the file significantly beyond budget. The planner must decide how to handle this -- likely by extracting reusable logic to a reference file (e.g., `references/body-generation-rules.md`) while keeping the Step 5 skeleton lean in SKILL.md.

**Primary recommendation:** Add Step 5 to SKILL.md with lean workflow instructions. Extract anti-hallucination rules, CEUS writing style rules, and bilingual LaTeX formatting examples to a new reference file loaded at Step 5 runtime. Update frontmatter to add `body_text` output type and `references/journals/ceus.md` to `references.required` (currently in `leaf_hints`).

### Locked Decisions
- Step 5 auto-continues in the same session after H3 confirmation -- no separate invocation needed
- When `.paper-refs/` is missing (Step 2.5 was skipped / MCP unavailable): continue generating, all citation positions use `[CITATION NEEDED]` placeholders -- do not block

- Before generating, Step 5 displays all H1 sections via `AskUserQuestion` (multiSelect) for user to select which sections to generate in this session
- LaTeX (.tex) output format -- CEUS-ready for direct submission
- Per-H1 files: one `.tex` file per H1 section (e.g., `introduction.tex`, `methods.tex`, `results.tex`)
- Directory: `{repo_path}/.paper-output/` -- alongside `.paper-refs/` in the experiment repo root
- Bilingual uses LaTeX comment format (`% --- Paragraph N ---` + `%` Chinese lines before English) per `references/bilingual-output.md`
- After all selected sections complete, auto-generate `references.bib` in `.paper-output/`
- Section-by-section generation: generate one H1 section, then wait for confirmation before the next
- Confirmation method: `AskUserQuestion` with options (Confirm / Modify / Skip) -- distinct from outline review which uses plain text
- Modification loop: user describes changes -> regenerate entire section -> re-display via AskUserQuestion -> loop until confirmed
- After all selected sections confirmed, generate `references.bib` and display completion summary
- `\cite{key}` inline embedding: all citations written as `\cite{citationkey}` directly in the paragraph
- Strict source boundary: `\cite{}` keys MUST only reference papers present in `.paper-refs/` files -- no citations from Claude's background knowledge
- Content that needs a citation but has no supporting `.paper-refs/` entry -> use `[CITATION NEEDED]` placeholder
- Auto-generate `references.bib`: read all `.paper-refs/*.md` files, extract all BibTeX blocks, deduplicate by citation key, write to `.paper-output/references.bib`
- Repo-derived claims (specific numbers, model names, dataset sizes, configurations): MUST include `[SOURCE: file:line]` annotation -- never stated without repo evidence
- Quantitative results without supporting repo data: use `[RESULTS NEEDED]` placeholder
- Specific metric values where the general metric is known but value is not: use `[EXACT VALUE: metric]`
- Background/context prose (general domain knowledge, problem framing): may be written as normal academic text -- no placeholder required

- Whether to include `\section{}`, `\subsection{}`, `\subsubsection{}` LaTeX heading commands in output files

### Recommended Modification Strategy

**Problem:** Adding Step 5 inline will push SKILL.md to ~500+ lines (Step 5 needs ~80-120 lines for generation rules, citation logic, bilingual formatting, `references.bib` generation, section selection UI, and confirmation loop).

**Solution: Extract to reference file.**

- CEUS writing style rules (tense, verbs, avoidance list)
- Bilingual LaTeX formatting template with examples

- Placeholder vocabulary (`[CITATION NEEDED]`, `[RESULTS NEEDED]`, `[EXACT VALUE: metric]`, `[SOURCE: file:line]`)

### Pattern: Step 5 Workflow Flow

```
Step 4 confirmed (H3 outline complete)
           |
           v
[Step 5: Body Generation]
           |
           v
Display all H1 sections via AskUserQuestion (multiSelect)
  -> User selects which sections to generate
           |
           v
FOR each selected H1 section (sequentially):
  1. Read relevant repo files (using repo-patterns mapping)
  2. Read .paper-refs/{section}.md (if exists) for citation keys
  3. Generate full section .tex content:
     - \section{}, \subsection{}, \subsubsection{} headings
     - Academic prose with CEUS style
     - \cite{key} inline (from .paper-refs only)
     - [SOURCE: file:line] annotations on repo-derived claims
     - [CITATION NEEDED] where no .paper-refs entry exists
     - [RESULTS NEEDED] / [EXACT VALUE: metric] for unknown data
     - Bilingual: % Chinese comment lines before each English paragraph
  4. Write to .paper-output/{section}.tex
  5. Display generated section to user
  6. AskUserQuestion: Confirm / Modify / Skip
     - Modify: user describes changes -> regenerate -> loop
     - Confirm: proceed to next section
     - Skip: move to next section without writing
END FOR
           |
           v
Auto-generate .paper-output/references.bib:
  1. Read all .paper-refs/*.md files
  2. Extract all ```bibtex ... ``` blocks
  3. Deduplicate by citation key (first occurrence wins)
  4. Write consolidated references.bib
           |
           v
Display completion summary
```

### Anti-Patterns to Avoid
- **Inline all rules in SKILL.md:** Adding 100+ lines of generation rules directly into the Workflow section will make the file unmanageable and exceed budget by 50%+

- **Blocking on missing .paper-refs/:** Decision is to continue with `[CITATION NEEDED]` placeholders, never block
- **Citing from background knowledge:** Only `.paper-refs/` citation keys are allowed; no `\cite{}` from Claude's training data

## Common Pitfalls

### Pitfall 1: SKILL.md Line Budget Explosion
**What goes wrong:** Adding Step 5 with all generation rules, citation logic, bilingual format, and `references.bib` generation inline pushes the file to 500+ lines.
**Why it happens:** Step 5 is the most complex step in the workflow -- it combines text generation, citation management, anti-hallucination rules, bilingual formatting, and file I/O.
**How to avoid:** Extract all rule definitions to `references/body-generation-rules.md`. Keep Step 5 in SKILL.md as a lean workflow skeleton (~40-50 lines) that references the rule file.
**Warning signs:** Step 5 draft exceeds 80 lines in SKILL.md.

### Pitfall 2: Confusing Outline Confirmation vs Body Confirmation Patterns
**What goes wrong:** Using plain text + "ok" for body section confirmation (the outline pattern) instead of AskUserQuestion with Confirm/Modify/Skip.
**Why it happens:** Steps 2-4 use plain text confirmation; it's tempting to copy that pattern for Step 5.
**How to avoid:** Phase 16 CONTEXT.md explicitly specifies AskUserQuestion for body section review. This is intentionally different from outline review because body sections are more complex and benefit from structured options.
**Warning signs:** Step 5 says "wait for user confirmation" without specifying AskUserQuestion.

### Pitfall 3: Forgetting the Section Selection Step
**What goes wrong:** Step 5 starts generating all sections without asking which ones to generate.
**Why it happens:** Steps 2-4 generate all sections at each level. Step 5 is different -- it generates selectively per user choice.
**How to avoid:** Step 5 MUST begin with AskUserQuestion (multiSelect) listing all H1 sections. User picks which sections to generate in this session.
**Warning signs:** No AskUserQuestion call at the start of Step 5.

### Pitfall 4: Citation Source Boundary Violation
**What goes wrong:** `\cite{key}` references papers not in `.paper-refs/` files, sourced from Claude's background knowledge.
**Why it happens:** Claude naturally wants to cite relevant papers it knows about.
**How to avoid:** The rule is absolute: `\cite{}` keys MUST come from `.paper-refs/` files. If a claim needs a citation but no `.paper-refs/` entry supports it, use `[CITATION NEEDED]`.
**Warning signs:** `\cite{}` keys in output that don't match any BibTeX entry in `.paper-refs/`.

### Pitfall 5: Missing `[SOURCE: file:line]` on Repo-Derived Claims
**What goes wrong:** Specific numbers, model names, dataset sizes stated as plain text without source annotation.
**Why it happens:** It's easy to write "the model uses 500 estimators" without citing the source file and line.
**How to avoid:** Every repo-derived specific claim MUST have `[SOURCE: file:line]` annotation. The claim classification rules in `body-generation-rules.md` make this explicit.
**Warning signs:** Specific numbers or configurations in output without `[SOURCE:]` annotations.

### Pitfall 6: references.bib Generation Before All Sections Confirmed
**What goes wrong:** Generating `references.bib` after each section instead of at the end.
**Why it happens:** Seems logical to generate incrementally.
**How to avoid:** CONTEXT.md specifies: "After all selected sections confirmed, generate `references.bib`." It runs once at the end, not per-section.
**Warning signs:** `references.bib` write operation inside the per-section loop.

### Example 5: references.bib Generation Logic

```
1. Read all files matching {repo_path}/.paper-refs/*.md
2. For each file:
   a. Extract all content between ```bibtex and ``` markers
   b. Parse each BibTeX entry to extract citation key
3. Deduplicate: if same citation key appears in multiple files,
   keep the first occurrence only
4. Write all unique entries to {repo_path}/.paper-output/references.bib
5. Display count: "Generated references.bib with N unique entries."
```

Source: Phase 16 CONTEXT.md decision on references.bib auto-generation.
