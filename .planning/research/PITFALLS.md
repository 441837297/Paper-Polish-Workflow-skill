# Pitfalls Research: v2.1 Team Agents Orchestration Layer

**Domain:** Adding parallel section-level processing and multi-Skill pipelines to an existing 13-Skill Claude Code suite for academic paper writing
**Researched:** 2026-03-19
**Confidence:** MEDIUM-HIGH (grounded in codebase analysis of all 13 existing Skills, verified Claude Code official docs on subagents and agent teams, community-reported bugs and workarounds, and academic research on multi-agent LLM consistency drift)

---

## Critical Pitfalls

Mistakes that cause architectural rewrites, corrupt user papers, or make the orchestration layer fundamentally unusable.

---

### Pitfall 1: Parallel Agents Writing to the Same Paper File Simultaneously

**Severity:** BLOCKING

**What goes wrong:**
The orchestrator splits a paper into H1/H2 sections and dispatches parallel agents (one per section) running the same Skill (e.g., `ppw:polish`). Multiple agents attempt to Edit the same `.tex` file concurrently. Claude Code has no file-locking mechanism for the Edit tool -- the last agent to write wins, silently overwriting all previous agents' edits. The user ends up with a paper where only one section is polished and all other sections revert to their original state, with no error message indicating data loss.

This is the single most dangerous pitfall because it causes **silent data loss** on the user's actual paper.

**Why it happens:**
- All 13 existing Skills operate on a single paper file (or a single section pasted in conversation). The `ppw:polish` Skill uses the Edit tool for in-place modification with `% [Polish] Original:` annotations. The `ppw:de-ai` Skill does the same with `% [De-AI] Original:` annotations. These Skills were designed for serial, single-agent execution.
- Claude Code's official docs explicitly warn: "Two teammates editing the same file results in overwrites." The race condition on `.claude.json` corruption (fixed in v2.1.61) demonstrated that parallel file writes without locking cause corruption even on Claude Code's own internal files.
- The natural mental model -- "split paper into sections, process each in parallel, merge results" -- hides the critical detail that "merge" requires explicit coordination. There is no automatic merge mechanism.
- Git worktrees (Claude Code's recommended isolation mechanism) are designed for code repositories, not for a single paper file that must be reassembled. Creating 6 worktrees for 6 sections of one file is architectural overkill and introduces its own merge complexity.

**How to avoid:**
1. **Section-to-file decomposition before dispatch**: the orchestrator must physically split the paper into separate temporary files (one per section) before spawning agents. Each agent owns exactly one file. After all agents complete, the orchestrator reassembles the sections into the original file. This is a hard requirement, not an optimization.
2. **File ownership contract in agent prompts**: each agent's spawn prompt must explicitly state: "You own `sections/03-methods.tex`. Do NOT read or write any other section file." This prevents agents from accidentally touching shared files.
3. **Sequential reassembly by the lead**: only the orchestrator (team lead / main session) performs the final merge. Agents return their results; they never write to the original paper file.
4. **Backup before dispatch**: the orchestrator must copy the original paper file to `[name]_backup_[timestamp].tex` before any agent is spawned. If the merge fails, the user can recover.

**Warning signs:**
- The orchestrator design says "agents edit the paper in-place" without a file decomposition step
- No backup mechanism exists before parallel dispatch
- Agent prompts do not specify file ownership boundaries
- Testing with 2+ parallel agents produces output where some sections are missing changes

**Phase to address:**
Phase 1 (Section Splitter design). The file decomposition and reassembly mechanism is a prerequisite for any parallel execution. Without it, nothing else in the orchestration layer is safe to build.

### Pitfall 2: Consistency Drift Across Parallel Section Outputs

**Severity:** BLOCKING

**What goes wrong:**
Parallel agents processing different sections of the same paper produce outputs with inconsistent terminology, tone, hedging levels, and style. Section 2 uses "urban heat island effect" while Section 4 uses "UHI phenomenon." Section 3 uses strong hedging ("Our results suggest") while Section 5 uses assertive claims ("This demonstrates"). The reviewer-simulation Skill would flag these as inconsistencies, but the orchestrator has already merged and delivered the output.

Academic papers demand strict internal consistency. A paper with visibly inconsistent style across sections immediately signals to reviewers that it was machine-generated by separate processes -- the exact outcome this tool suite is designed to prevent.

**Why it happens:**
- Each parallel agent runs in its own context window with no shared state. Agent A processing the Introduction has no visibility into Agent B's terminology choices in the Methods section.
- The academic research on "agent drift" (arXiv 2601.04170, January 2026) formally documents this phenomenon: "semantic drift" causes "progressive deviation from original intent while remaining syntactically valid." Even with identical system prompts, parallel LLM instances produce measurably different stylistic outputs.
- The existing Skills (polish, translation, de-ai) handle consistency through sequential processing: when `ppw:polish` processes a long paper, it processes sections sequentially and "maintains terminology consistency across sections" (from the polish Skill's Step 3). Parallelization breaks this sequential consistency mechanism.
- Geographic and urban science papers are especially vulnerable: domain terminology has multiple valid forms ("built environment" vs. "built-up area" vs. "urban fabric"), and each agent may independently choose a different valid form.
- Expression pattern reference files provide a menu of alternatives (e.g., introduction-and-gap.md offers multiple ways to state a research gap). Different agents, given the same reference menu, will independently select different options.

**How to avoid:**
1. **Terminology anchor document**: before dispatching parallel agents, the orchestrator extracts a terminology glossary from the existing paper (key terms, abbreviations, preferred phrasings) and includes it in every agent's spawn prompt. This anchors all agents to the same terminology decisions.
2. **Style anchor from first section**: process the first section (usually Introduction) sequentially. Extract the resulting style profile (hedging level, citation format, tense usage, terminology choices) and inject it as a constraint into all subsequent parallel agents' prompts.
3. **Mandatory post-merge consistency review**: after reassembling sections, run an automated consistency check as a dedicated pass. This check compares: (a) terminology variants across sections, (b) hedging language distribution, (c) tense consistency, (d) abbreviation usage (defined before first use, used consistently after). This is the "automatic consistency review" listed in PROJECT.md's target features.
4. **Shared glossary file**: write the terminology anchor to a shared file that all agents read (read-only). Since all agents are only reading this file (not writing), there is no race condition.

**Warning signs:**
- Agent spawn prompts contain no terminology or style constraints
- No consistency review step exists between parallel processing and output delivery
- Testing with 3+ parallel sections reveals different abbreviation expansions or hedging levels
- The orchestrator merges sections and immediately delivers without any post-processing

**Phase to address:**
Phase 2 (Single-Skill Parallel Mode). The terminology anchor and style profiling must be part of the parallel dispatch design. The consistency review should be built in Phase 4 (Consistency Review) but the architecture must accommodate it from Phase 2.

### Pitfall 3: Skill Invocation Fails Inside Subagents/Teammates

**Severity:** BLOCKING

**What goes wrong:**
The orchestrator spawns a subagent or teammate and instructs it to "run `ppw:polish` on this section." The agent either:
(a) Cannot find the Skill because Skills are not automatically inherited by subagents (they must be explicitly listed in the `skills` frontmatter field),
(b) Finds the Skill but cannot load its required reference files because the agent's working directory or file access differs from the main session,
(c) Attempts to invoke the Skill via a Bash command (`claude /ppw:polish`) instead of using it as a prompt instruction, causing an error, or
(d) Loads the Skill but the teammate's restricted tool set (20 tools vs. 25 for subagents) prevents the Skill from functioning -- teammates cannot use the Agent tool, which means a Skill that internally delegates to another agent will fail silently.

**Why it happens:**
- Claude Code official docs state: "Subagents don't inherit skills from the parent conversation; you must list them explicitly." The `skills` frontmatter field injects Skill content into the agent's context at startup. Without this, the Skill is invisible.
- When Skills are injected via the `skills` field, the "full content of each skill is injected" -- meaning the entire SKILL.md is loaded into the agent's context. For ppw Skills that are ~300 lines each and require 2-6 reference file loads, this adds significant context consumption per agent.
- Reference files at `references/expression-patterns/*.md` and `references/anti-ai-patterns/*.md` must be accessible via the Read tool. Subagents and teammates inherit Read access by default, but if the orchestrator spawns agents in git worktrees (the `isolation: worktree` option), the reference files must exist in the worktree too.
- The GitHub issue #32731 documents that "teammates have fewer tools than subagents and cannot spawn anything." If the orchestrator Skill (`ppw:team`) needs to spawn further sub-agents from within a teammate, it will fail.
- Community reports note that "skills invoked by subagents incorrectly appearing in the main session after compaction" is a known bug, suggesting the Skill-in-subagent path is still being stabilized.

**How to avoid:**
1. **Use subagents (Agent tool), not agent teams (TeamCreate)**: for this project, subagents are the better fit. The orchestrator does not need inter-agent communication -- each agent processes its section independently and returns results to the lead. Subagents have fewer restrictions (25 tools vs. 20 for teammates) and do not require the experimental `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` flag.
2. **Pre-load Skills via the `skills` field in agent definitions**: create a dedicated agent definition at `.claude/agents/ppw-section-worker.md` that lists the Skills it needs:
   ```yaml
   skills:
     - ppw-polish
     - ppw-de-ai
     - ppw-translation
   ```
   This injects the Skill content at spawn time. However, this loads ALL listed Skills into context, which is expensive. An alternative is to spawn the agent with only the specific Skill needed for the current operation.
3. **Do NOT use git worktrees for isolation**: worktrees isolate the entire repository. Since ppw Skills need access to `references/` files in the main working directory, worktree isolation creates unnecessary complexity. Instead, use the file ownership contract (Pitfall 1) for isolation.
4. **Embed Skill instructions directly in the agent prompt**: instead of relying on the Skills system, the orchestrator can construct the agent's prompt by reading the relevant SKILL.md and reference files, then passing the concatenated content as the agent's task prompt. This is more explicit and avoids the Skill discovery/loading chain, but consumes more tokens per agent.
5. **Test Skill invocation in a subagent before building the orchestrator**: the very first implementation task should be: "spawn one subagent, have it run ppw:polish on one section, verify the output matches what ppw:polish produces in the main session." If this does not work, the entire orchestration architecture needs rethinking.

**Warning signs:**
- Agent produces generic polishing instead of following the ppw:polish Skill's specific workflow (reference loading, LaTeX annotations, anti-AI screening)
- Error messages about missing reference files in agent output
- Agent attempts to invoke Skills as Bash commands
- Different output quality between main-session Skill execution and agent-delegated Skill execution

**Phase to address:**
Phase 1 (Architecture). The Skill invocation mechanism must be validated as a proof-of-concept before any orchestration logic is built. This is a go/no-go gate.

### Pitfall 4: Context Window Exhaustion in Section-Processing Agents

**Severity:** BLOCKING

**What goes wrong:**
Each parallel agent must load: (a) the Skill content (~300 lines / ~2,400 tokens), (b) required reference files (expression-patterns + anti-AI patterns = 3-6 files, ~8,000-15,000 tokens), (c) the section text to process (~2,000-8,000 tokens for a typical paper section), and (d) generate output (~3,000-6,000 tokens for polished + annotated text). Total per agent: 15,000-31,000 tokens minimum.

If the orchestrator uses the default model (inherited from main session), agents run on the same model. With Sonnet's 200K context or even Opus's 200K default, each agent has plenty of room. But the per-agent cost multiplied by 5-8 parallel agents means total token consumption is 5-8x a serial run. More critically, if agents are spawned with `model: haiku` to reduce cost, Haiku's smaller effective context and lower capability may produce significantly degraded academic polishing quality.

The real danger is not hard context limits but quality degradation as context fills. If an agent loads 6 reference files, the Skill instructions, the terminology anchor, AND the section text, the model's attention is spread thin and output quality drops.

**Why it happens:**
- The existing Skills were designed for the main session's context, where they are the only active task. In the orchestrated model, each agent is a fresh context that must bootstrap all necessary knowledge from scratch.
- The ppw:polish Skill loads "all three anti-AI pattern leaves proactively" plus the appropriate expression pattern leaf plus the journal template. That is 4-5 reference files loaded before any work begins.
- The v2.0 PITFALLS.md (Pitfall 1) already documented context exhaustion for repo-to-paper. The same problem recurs at a smaller scale for each parallel agent processing a section.
- Cost scaling: Anthropic's documentation notes agent teams cost "roughly 3-7x more tokens than a single session." For section-level parallelization with 6 sections, the multiplier is approximately 6x for the processing alone, plus orchestration overhead.

**How to avoid:**
1. **Minimize reference loading per agent**: instead of loading all reference files, the orchestrator pre-determines which specific leaf reference file each agent needs based on section type (introduction -> introduction-and-gap.md, methods -> methods-and-data.md). Include only the relevant leaf in the agent's prompt context, not the full overview + all leaves.
2. **Pre-extract reference content into the prompt**: rather than having each agent Read reference files at runtime (each Read consumes tool-call overhead), the orchestrator reads the necessary reference files once and injects the relevant content directly into each agent's spawn prompt. This saves tool-call tokens and makes the agent's context deterministic.
3. **Set `maxTurns` on agents**: prevent runaway agents from filling their context with repeated tool calls. A polish operation on one section should complete in 5-10 turns. Set `maxTurns: 15` as a safety cap.
4. **Use Sonnet for section workers, not Haiku**: Haiku is appropriate for exploration/search subagents but NOT for academic text polishing. The quality difference is significant for nuanced tasks like hedging calibration, anti-AI vocabulary screening, and domain-specific expression selection. Sonnet provides the best cost/quality tradeoff for section processing.
5. **Budget tokens per agent**: calculate expected token consumption before dispatching. If total estimated consumption (agents x per-agent cost) exceeds the user's expected cost tolerance, warn before proceeding.

**Warning signs:**
- Agents produce noticeably lower quality output than the same Skill run in the main session
- Agent outputs lack anti-AI screening or expression pattern application (signs of context-starved processing)
- Token cost for a parallelized run is 10x+ the serial equivalent
- Agents time out or hit maxTurns limits on sections that should be straightforward

**Phase to address:**
Phase 2 (Single-Skill Parallel Mode). Reference loading optimization is part of the parallel dispatch design. Cost estimation should be built into Phase 3 (Observable Progress).

### Pitfall 5: workflow-memory.json Race Condition Under Parallel Writes

**Severity:** Moderate (data corruption, not paper corruption)

**What goes wrong:**
All 13 existing Skills append to `.planning/workflow-memory.json` as their first workflow step ("Record workflow"). When the orchestrator runs `ppw:polish` on 6 sections in parallel, all 6 agents attempt to read, modify, and write `workflow-memory.json` simultaneously. Without file locking, this causes:
- Lost entries (agent A reads file, agent B reads same version, both append and write, B's write overwrites A's append)
- JSON corruption (partial writes produce invalid JSON)
- Entry count exceeding the 50-entry cap because multiple agents check the count against the same snapshot

This mirrors the `.claude.json` corruption bug (GitHub issue #28847) that was fixed in v2.1.61.

**Why it happens:**
- The Workflow Memory convention (skill-conventions.md) specifies: "append one entry to `.planning/workflow-memory.json`" and "if log length >= 50, drop the oldest entry before appending." This read-modify-write cycle is inherently unsafe under concurrent access.
- Every Skill has this code. The grep shows 26 occurrences of `workflow-memory.json` across all 13 SKILL.md files (2 per Skill: one for reading in Step 0, one for writing in Step 1).
- The Workflow Memory system was designed in v2.0 for serial, one-Skill-at-a-time execution. Parallel execution was not considered.

**How to avoid:**
1. **Orchestrator handles workflow memory, not individual agents**: when the orchestrator dispatches agents, it records a single workflow memory entry for the orchestrated operation (e.g., `{"skill": "ppw:team:polish", "ts": "...", "sections": 6}`). Individual section agents skip Step 0 (pattern detection) and the workflow memory append entirely.
2. **Agent-specific Skill variants**: create lightweight "agent mode" instructions that skip the workflow memory read/write. The orchestrator's spawn prompt includes: "Skip Step 0 (Workflow Memory Check) and skip the workflow memory recording in Step 1. The orchestrator handles workflow tracking."
3. **Never let agents write to shared JSON files**: this is a general principle. Any file that multiple agents might write to must be owned exclusively by the orchestrator.

**Warning signs:**
- `workflow-memory.json` contains duplicate timestamps or corrupted JSON after a parallel run
- Pattern detection (Step 0) fires incorrectly because the memory log has garbled entries
- The 50-entry cap is exceeded with 60+ entries after a parallel run

**Phase to address:**
Phase 2 (Single-Skill Parallel Mode). The orchestrator must override the default Skill behavior for workflow memory from day one.

### Pitfall 6: Multi-Skill Pipeline Stage Ordering and Error Propagation

**Severity:** Moderate (wasted compute, confusing error states)

**What goes wrong:**
The multi-Skill pipeline mode chains Skills (e.g., translate -> polish -> de-ai). When a section fails at the "polish" stage (e.g., the agent hits maxTurns, the reference file is missing, or the output is truncated), the pipeline has three bad options:
(a) Skip the failed section and continue with de-ai on the remaining sections -- resulting in an inconsistent paper where some sections have 3 passes and one has only 1.
(b) Halt the entire pipeline -- wasting all work done by other sections that completed successfully.
(c) Retry the failed section -- but there is no retry mechanism, and the context from the failed attempt is lost.

In all three cases, the user sees a partially-processed paper with no clear indication of which sections completed which stages.

**Why it happens:**
- Claude Code's subagent model is fire-and-forget: the Agent tool spawns a subagent, it runs to completion (or maxTurns), and returns a result string. There is no built-in checkpoint, retry, or partial-success mechanism.
- The pipeline assumes each stage produces a well-formed output that the next stage can consume. But Skills can fail silently (producing low-quality output instead of erroring) or loudly (hitting tool access errors, reference loading failures).
- Error detection in LLM outputs is fundamentally harder than in traditional software: there is no "exit code." A polish agent that runs but produces generic output (because it could not load reference files) returns successfully with bad content.

**How to avoid:**
1. **Stage-gate validation between pipeline stages**: after each stage completes, the orchestrator validates the output before passing it to the next stage. Validation checks: (a) output file exists and is non-empty, (b) output length is within expected range (not truncated), (c) stage-specific markers exist (e.g., `% [Polish] Original:` annotations for polish stage).
2. **Per-section status tracking**: the orchestrator maintains a status file (`sections_status.json`) recording each section's current stage, completion status, and any error messages. This file is the single source of truth for pipeline progress.
3. **Partial retry, not full restart**: if Section 3 fails at the polish stage, the orchestrator retries only Section 3's polish stage using the translate-stage output as input. Other sections continue to the de-ai stage independently.
4. **Graceful degradation with user notification**: if retry also fails, the orchestrator delivers the paper with a clear annotation: "Section 3: translation complete, polish FAILED (reason), de-ai skipped. Run `ppw:polish` manually on this section."
5. **Set output expectations per stage**: each stage's agent prompt should specify the expected output format. The orchestrator can then validate structurally (annotations present, file written, word count reasonable) even if it cannot validate content quality.

**Warning signs:**
- Pipeline delivers output with no record of which sections completed which stages
- A failed section causes the entire pipeline to hang or abort
- User receives a paper where some sections are polished and de-AI'd while others are only translated, with no explanation
- No retry mechanism exists; failures require restarting the entire pipeline

**Phase to address:**
Phase 3 (Multi-Skill Pipeline Mode). The stage-gate validation and status tracking are core to the pipeline architecture.

### Pitfall 7: LaTeX Annotation Conflicts from Multiple Skill Passes

**Severity:** Moderate (output quality, not data loss)

**What goes wrong:**
In the multi-Skill pipeline (translate -> polish -> de-ai), each Skill adds its own LaTeX comment annotations:
- Translation adds `% Translated from: [source]`
- Polish adds `% [Polish] Original: <text>`
- De-AI adds `% [De-AI] Original: <text>`

When run in pipeline, the Polish agent encounters translation annotations and may either: (a) treat them as content to polish (incorrect), (b) strip them and add its own (losing translation traceability), or (c) become confused by the mixed annotation format and produce degraded output.

Furthermore, the de-ai Skill's Step 1 says "if existing `% [De-AI] Original:` annotations found, clean them up before adding new ones." In a pipeline, the de-ai agent might also clean up the Polish annotations because the cleanup regex is not strictly scoped to its own annotation format.

**Why it happens:**
- Each Skill's annotation format was designed for standalone use. The Skills explicitly handle their OWN previous annotations (polish cleans old polish annotations, de-ai cleans old de-ai annotations) but were never tested with another Skill's annotations present in the same file.
- The "clean up before adding new" pattern in both polish and de-ai Skills uses line-matching regexes. If the regex is too broad (`^% \[.*\] Original:`), it will match all annotation types. If it is correctly scoped (`^% \[Polish\] Original:` and `^% \[De-AI\] Original:` respectively), annotations from other Skills are preserved.
- In the parallel-then-pipeline scenario, sections may arrive at the de-ai stage with fresh polish annotations that should be preserved for user traceability.

**How to avoid:**
1. **Verify annotation regex scoping**: audit each Skill's cleanup regex to confirm it only matches its own annotation format. The current Skills use `^% \[Polish\] Original:` and `^% \[De-AI\] Original:` respectively, which are correctly scoped. Document this contract explicitly for the pipeline.
2. **Pipeline-aware annotation strategy**: in pipeline mode, the orchestrator instructs agents to PRESERVE annotations from previous stages. Add to each agent's prompt: "Do NOT remove or modify lines matching `% [Translation]`, `% [Polish]`, or `% [De-AI]` annotations from previous pipeline stages."
3. **Annotation accumulation as a feature**: the layered annotations create a full edit history -- the user can see the original Chinese, the translated English, the polished version, and the de-AI'd version. Document this as a feature, not a bug.
4. **Clean-slate option**: if the user wants a clean final output, the orchestrator offers a "strip all annotations" pass after pipeline completion, removing all `% [*] Original:` lines in one operation.

**Warning signs:**
- De-AI output is missing Polish annotations that should have been preserved
- Agents produce output with scrambled annotation ordering
- Cleanup regex in SKILL.md uses a broad pattern like `% \[.*\]` instead of the specific annotation type

**Phase to address:**
Phase 3 (Multi-Skill Pipeline Mode). Annotation preservation must be specified in the pipeline's inter-stage contract.

### Pitfall 8: User Cannot Intervene During Long-Running Parallel Operations

**Severity:** Moderate (UX, wasted tokens)

**What goes wrong:**
The orchestrator dispatches 6 parallel agents to polish 6 sections. After seeing the first completed section, the user realizes the agents are using the wrong journal style or targeting the wrong hedging level. But the other 5 agents are already running (or have already completed) and there is no way to stop them, change the configuration, and restart. The user must wait for all agents to finish, discard the results, and restart the entire operation.

With agent teams, token costs are 3-7x a single session. Wasting a full parallel run because the configuration was wrong costs the user significant tokens and time.

**Why it happens:**
- Claude Code's background subagents "auto-deny anything not pre-approved" for tool permissions. But there is no mechanism to send a "stop" or "reconfigure" signal to a running subagent.
- The orchestrator's "observable progress display with intervention points" (from PROJECT.md target features) implies the ability to intervene between stages, but not WITHIN a stage. Once agents are dispatched, the current stage runs to completion.
- Foreground subagents block the main conversation until complete. Background subagents run independently but cannot receive messages (that requires agent teams, which are experimental and have their own issues).
- The PROJECT.md specifies "intervention points between stages" for multi-Skill pipelines, but says nothing about intervention within a single parallel dispatch.

**How to avoid:**
1. **Preview-first pattern**: before dispatching all sections in parallel, the orchestrator processes ONE section first (the shortest or most representative) and presents the result for user approval. Only after approval does it dispatch the remaining sections. This is analogous to the existing Skill convention of "two-phase workflow" (analyze -> confirm -> apply) but at the orchestration level.
2. **Batch size control**: instead of dispatching all sections at once, dispatch in batches of 2-3. After each batch completes, present results and ask whether to continue with the same settings. This trades some parallelism for user control.
3. **Clear cost estimate before dispatch**: before starting, the orchestrator estimates: "This will process 6 sections with ppw:polish. Estimated token cost: ~X tokens (~$Y). Proceed?" This sets expectations and gives the user an informed abort point.
4. **Cancel mechanism via the lead**: if the orchestrator detects that the user wants to cancel (via the main session), it can avoid spawning additional agents. Already-running agents will complete their current turn but no new ones will start.

**Warning signs:**
- All sections are dispatched simultaneously with no preview
- No cost estimate is presented before parallel dispatch
- User has no way to abort after seeing the first section's result
- The orchestrator design has "intervention points between stages" but not within the parallel dispatch of a single stage

**Phase to address:**
Phase 3 (Observable Progress Display). The preview-first pattern and batch control are core UX features of the orchestration layer.

### Pitfall 9: Orchestrator Skill Exceeds Context Budget by Loading All Sub-Skill Instructions

**Severity:** Moderate (performance, cost)

**What goes wrong:**
The `ppw:team` orchestrator Skill needs to know how to invoke each of the 13 sub-Skills. If it loads all 13 SKILL.md files (~4,500 lines total, ~36,000 tokens) into its context to understand their workflows, the orchestrator's own context is severely constrained before any actual work begins. Add the paper file, reference files for section splitting, and the terminology anchor document, and the orchestrator may exhaust 40-50% of its context before spawning a single agent.

**Why it happens:**
- The orchestrator needs to understand each Skill's input requirements, output contract, and reference dependencies to correctly construct agent spawn prompts. The natural approach is to read each Skill's documentation.
- The existing Skill convention mandates "on-demand loading" -- Skills load references only when needed. But the orchestrator needs a meta-view of ALL Skills to route sections correctly.
- The ~300-line Skill budget is designed for single-Skill sessions. An orchestrator Skill that coordinates 13 other Skills is architecturally different from any existing Skill.

**How to avoid:**
1. **Skill registry reference file**: create `references/skill-registry.md` containing a compact summary of each Skill: name, input format, output format, required references, and key constraints. The orchestrator loads this single file (~100-150 lines) instead of 13 individual SKILL.md files. This is the same "structured summary instead of raw content" pattern from v2.0's Pitfall 1.
2. **Orchestrator loads only the Skills it needs**: if the user requests "polish all sections," the orchestrator loads only the ppw:polish Skill summary, not all 13. The registry file enables this selective loading.
3. **Skill invocation templates**: pre-build prompt templates for common operations (e.g., "polish section," "translate section," "de-ai section") stored in `references/orchestration/templates/`. The orchestrator reads the appropriate template and fills in section-specific details, rather than reasoning from the full Skill documentation each time.
4. **The orchestrator Skill itself must be lean**: the ppw:team SKILL.md should focus on orchestration logic (section splitting, dispatch, merge, consistency review) and delegate Skill knowledge to reference files. Target 250-300 lines for the orchestrator Skill itself.

**Warning signs:**
- The orchestrator Skill reads 5+ SKILL.md files before starting any work
- The orchestrator's spawn prompts are vague or generic (signs of context-starved instruction construction)
- The ppw:team SKILL.md exceeds 400 lines because it embeds sub-Skill instructions inline
- Total token consumption for the orchestrator session alone exceeds 50K tokens before any agents are spawned

**Phase to address:**
Phase 1 (Architecture). The skill registry and invocation templates are architectural prerequisites.

### Pitfall 10: Agent Teams Experimental Status and Platform Instability

**Severity:** Moderate (reliability risk, not architectural)

**What goes wrong:**
The orchestrator is built on Claude Code's Agent Teams feature, which is experimental and disabled by default. Known issues as of March 2026 include:
- TeamCreate spawns teammates that silently exit (GitHub #34614, v2.1.76)
- Teammates lose tool access in Delegate Mode (GitHub #24073)
- No session resumption with in-process teammates (`/resume` does not restore teammates)
- Subagent-created teams persist on disk and block future team creation (GitHub #32730)
- Split-pane mode not supported on Windows Terminal (the user's platform)
- One team per session limitation

If the orchestrator depends on Agent Teams and the feature regresses or changes behavior in a Claude Code update, the entire orchestration layer breaks.

**Why it happens:**
- Agent Teams shipped February 5, 2026 -- barely 6 weeks old. Experimental features have unstable APIs and known bugs.
- The user is on Windows 11 with bash shell. Split-pane mode (tmux/iTerm2) is not natively available. In-process mode works but becomes "noisy with 3+ agents."
- Claude Code ships frequent updates (the `.claude.json` fix was v2.1.61, TeamCreate bug is v2.1.76 -- 15 minor versions in a short span). Any update could change agent behavior.

**How to avoid:**
1. **Build on subagents (Agent tool), not agent teams (TeamCreate)**: subagents are a stable, non-experimental feature. They support parallel execution (up to 10 concurrent), run in isolated context windows, and return results to the orchestrator. The only limitation vs. agent teams is no inter-agent messaging -- but our use case does not require it (agents process sections independently).
2. **Abstract the dispatch mechanism**: the orchestrator should have a dispatch abstraction layer that can use either subagents or agent teams. Build v2.1 on subagents; add agent teams as an optional upgrade path when the feature stabilizes.
3. **No dependency on split-pane display**: design the progress display for in-process/inline mode. The user is on Windows Terminal where split panes are not supported. Progress should be shown via the main session's conversation output.
4. **Test on current Claude Code version before each release**: since Claude Code updates frequently, validate the orchestrator's agent spawning works correctly before publishing any update.

**Warning signs:**
- The orchestrator design requires `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
- Progress display depends on tmux split panes
- Agent spawning occasionally fails silently with no error message
- A Claude Code update breaks the orchestrator without any code changes on our side

**Phase to address:**
Phase 1 (Architecture). The choice between subagents and agent teams is the foundational architectural decision and must be made first.

## Minor Pitfalls

Mistakes that cause inconvenience or suboptimal results but are easily corrected.

---

### Pitfall 11: Section Splitter Misidentifies Paper Structure

**What goes wrong:**
The section splitter incorrectly parses the paper's heading hierarchy. Common failures: (a) confusing LaTeX `\section{}` with `\subsection{}`, (b) splitting at `\paragraph{}` level (too granular), (c) failing to recognize Chinese headings in bilingual drafts, (d) including preamble/bibliography as sections to process, (e) splitting mid-paragraph when a heading is commented out.

**How to avoid:**
1. Parse only `\section{}` and `\subsection{}` markers for LaTeX files. For Markdown files, parse only `#` and `##` headings.
2. Exclude preamble (everything before `\begin{document}`) and bibliography (`\begin{thebibliography}` or `\bibliography{}`) from the section list.
3. Present the detected section list to the user for confirmation before dispatching agents.
4. Support manual section boundaries: allow the user to specify "split at lines 10, 45, 89, 130" if automatic detection fails.

**Phase to address:** Phase 1 (Section Splitter). Include user confirmation of detected sections.

---

### Pitfall 12: Bilingual Output Doubling in Orchestrated Mode

**What goes wrong:**
Each parallel agent produces bilingual output (English + Chinese blockquotes) as per the Skill's default behavior. The orchestrator then merges 6 sections, each with bilingual comparison, resulting in an extremely long conversation output that the user must scroll through. The bilingual comparison that is useful for one section becomes overwhelming when multiplied by 6.

**How to avoid:**
1. Disable bilingual conversation output in agent mode. Agents write results to files only.
2. The orchestrator produces a single consolidated bilingual comparison after merge, not per-section.
3. Offer bilingual output as a post-merge optional step, not a per-agent default.

**Phase to address:** Phase 2 (Single-Skill Parallel Mode). Agent prompts must include bilingual output suppression.

---

### Pitfall 13: Orchestrator Does Not Respect Existing Skill Interaction Modes

**What goes wrong:**
The orchestrator dispatches a Skill in `direct` mode (appropriate for parallel batch processing), but the Skill's default mode is `guided` or `interactive`. The agent follows the Skill's default mode and starts asking questions via AskUserQuestion -- but as a background subagent, AskUserQuestion calls fail (background subagents auto-deny clarifying questions). The agent proceeds with incorrect defaults or stalls.

**How to avoid:**
1. The orchestrator MUST set `direct` mode explicitly in every agent's spawn prompt: "Run in direct mode. Do not ask questions. Use the following settings: [journal, section type, etc.]"
2. Pre-resolve all Skill parameters (journal, section type, glossary path) in the orchestrator before dispatch. Agents should never need to ask the user anything.
3. Set agent permission mode to `bypassPermissions` so tool access is never blocked by permission prompts.

**Phase to address:** Phase 2 (Single-Skill Parallel Mode). Part of the agent spawn prompt construction.

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Using Agent Teams instead of subagents | Inter-agent messaging, shared task list | Dependency on experimental feature; Windows incompatibility; frequent breakage | Never for v2.1 -- use subagents. Revisit when Agent Teams exits experimental. |
| Embedding section-specific reference content in agent prompts instead of having agents Read files | Deterministic context, no tool-call overhead | Prompt size grows with reference updates; changes to reference files require orchestrator update | Acceptable for small reference files (< 50 lines). Use Read for larger references. |
| Skipping consistency review to save tokens | 20-30% fewer tokens per orchestrated run | Inconsistent terminology, hedging, and style across sections that reviewers will flag | Never for multi-section parallel operations. Acceptable only for single-section runs. |
| Hardcoding section split patterns instead of user confirmation | Faster workflow, no question asked | Fails on non-standard paper structures; frustrates users with unusual heading conventions | Only for well-formatted LaTeX with standard `\section{}` commands. Always confirm for Markdown or mixed formats. |
| Processing all sections with the same reference leaf | Simpler orchestrator logic; one reference set per dispatch | Introduction processed with methods-and-data expression patterns, producing inappropriate phrasing | Never -- section-type detection and reference routing is a core orchestrator responsibility |

## Integration Gotchas

Common mistakes when connecting the orchestration layer to the existing Skill ecosystem.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Orchestrator -> ppw:polish | Spawning agent without anti-AI reference files, producing output that fails de-AI screening | Pre-load all three anti-AI leaves into the agent prompt. The polish Skill loads them "proactively" (always), so the agent must too. |
| Orchestrator -> ppw:translation | Agent attempts to create `_en.tex` and `_bilingual.tex` output files with conflicting names when multiple section agents run in parallel | Each agent writes to a section-specific filename (e.g., `section_03_methods_en.tex`). The orchestrator handles final assembly. |
| Orchestrator -> ppw:de-ai | Agent runs detection phase and presents interactive selection (Step 4: "Fix all High Risk") but no user is available to respond | In orchestrated mode, instruct the agent to automatically apply "Fix all High Risk + Medium Risk" without user selection. |
| Orchestrator -> workflow-memory.json | Multiple agents append to the same JSON file simultaneously | Orchestrator records a single entry; agents skip workflow memory entirely. |
| Orchestrator -> ppw:reviewer-simulation | Spawning parallel reviewer agents per section produces section-level reviews but misses cross-section concerns (argument flow, evidence progression) | Run reviewer-simulation AFTER merge, on the complete paper, in the main session. It is not suitable for parallel section-level execution. |

## Performance Traps

Patterns that work at small scale but fail as paper size or pipeline depth grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Loading all 13 Skill definitions into the orchestrator | Works for a "show me what's available" command | Skill registry reference file with compact summaries | Orchestrator context fills before dispatching any agents |
| Spawning one agent per paragraph instead of per section | Works for a 3-paragraph abstract | Section-level granularity (H1 or H2 sections) | Papers with 30+ paragraphs; overhead per agent exceeds processing benefit |
| Running consistency review on every intermediate pipeline stage | Works for a 2-stage pipeline | Consistency review only after final merge | 3+ stage pipelines where review overhead exceeds processing time |
| Using Opus for section-processing agents | Produces highest quality output | Use Sonnet for section processing; reserve Opus for consistency review and orchestration | Cost becomes prohibitive with 6+ parallel Opus agents |
| Retrying failed agents without diagnosing the failure cause | Works when the failure is transient (network blip) | Diagnose failure type before retry; cap at 1 retry per section | Systematic failures (missing reference file) that will fail on every retry |

## UX Pitfalls

Common user experience mistakes in multi-agent orchestration.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Showing raw agent output in the terminal during parallel processing | Information overload; 6 agents producing interleaved output is unreadable | Show progress summary only: "Section 3/6 complete. Sections 4,5,6 in progress." Show detailed output per section only on request. |
| No progress indication during long-running parallel operations | User thinks the system is frozen; may abort and lose work | Display a live progress indicator: section name, current stage, elapsed time. Update as each agent completes. |
| Dumping the full merged paper into conversation after completion | Conversation context consumed by the output; user must scroll to find changes | Write merged output to file. In conversation, show only a summary: "6 sections polished. Changes: 47 edits (12 structural, 23 expression, 12 anti-AI). Output written to `paper_polished.tex`." |
| Requiring the user to understand subagent architecture | "What is a subagent? What is a team lead?" | Abstract away the agent mechanism entirely. User says "polish all sections in parallel" and sees progress + results. The orchestration is invisible infrastructure. |
| No way to inspect individual section results before merge | User must accept the full merged output or reject everything | After agents complete and before merge, present a section-by-section summary with an option to re-run specific sections. |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **Section splitter:** Often missing preamble/bibliography exclusion -- verify it does not dispatch agents on `\usepackage` blocks or `\begin{thebibliography}`
- [ ] **Parallel dispatch:** Often missing file ownership contracts -- verify each agent's prompt explicitly names its owned file and prohibits writing elsewhere
- [ ] **Consistency review:** Often missing terminology extraction -- verify it compares terminology across sections, not just formatting
- [ ] **Multi-Skill pipeline:** Often missing inter-stage validation -- verify a gate check exists between each stage that validates output format and completeness
- [ ] **Progress display:** Often missing failure reporting -- verify the display shows which sections failed and why, not just which succeeded
- [ ] **Agent spawn prompts:** Often missing direct-mode enforcement -- verify agents are instructed to skip AskUserQuestion and use pre-resolved parameters
- [ ] **Merge step:** Often missing annotation preservation -- verify all LaTeX annotations from all pipeline stages survive the merge
- [ ] **Backup mechanism:** Often missing entirely -- verify the original paper is backed up before any agent is dispatched
- [ ] **Cost estimation:** Often missing user notification -- verify the orchestrator warns the user about estimated token cost before parallel dispatch
- [ ] **Workflow memory:** Often missing parallel-safety -- verify agents skip workflow-memory.json writes and the orchestrator records a single entry

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| P1: Parallel file write corruption | HIGH | Restore from backup file. If no backup, use git history to recover. Re-run the entire parallel operation with file decomposition. |
| P2: Consistency drift across sections | MEDIUM | Run the post-merge consistency review. Apply terminology normalization pass. May require re-processing 1-2 sections with corrected anchor. |
| P3: Skill invocation failure in agents | MEDIUM | Diagnose which Skill component failed (loading, reference access, tool access). Fix the agent definition or spawn prompt. Re-run failed sections only. |
| P4: Context exhaustion in agents | LOW-MEDIUM | Reduce reference loading. Pre-extract content into prompts. Re-run with optimized agent configuration. |
| P5: workflow-memory.json corruption | LOW | Delete the file and recreate as `[]`. No impact on paper content -- only affects pattern detection convenience. |
| P6: Pipeline error propagation | MEDIUM | Inspect per-section status. Re-run failed sections from the last successful stage. Merge selectively. |
| P7: Annotation conflicts | LOW | Audit cleanup regexes. Re-run the pipeline stage that stripped annotations, with corrected prompts. |
| P8: No user intervention possible | HIGH | Discard results from the unwanted run. Token cost is non-recoverable. Restart with corrected settings using preview-first pattern. |
| P9: Orchestrator context bloat | LOW | Create skill registry. Reload orchestrator with lean context. No impact on user data. |
| P10: Platform instability | MEDIUM | Switch from agent teams to subagents (or vice versa). Abstract dispatch layer makes this a configuration change, not a rewrite. |
